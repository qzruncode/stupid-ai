# Full Scan — 全景治理扫描

一次跑完 repo-guardian 的所有维度，输出一份完整的治理报告。

## 工作步骤

### 1. 先扫全貌

读 `README`、`package.json`、`tsconfig`，确定项目类型和技术栈，建立模块地图。

### 2. 按顺序执行各维度扫描

| 步骤 | 做什么 | 产出 |
|---|---|---|
| 1 | 架构审查 | 当前架构地图 + 架构问题 |
| 2 | 依赖审计 | 依赖健康报告 + 替代建议 |
| 3 | 代码质量扫描 | 质量问题清单 |
| 4 | 性能审计 | 性能瓶颈清单 |
| 5 | 设计审查 | 前端质量报告（如果项目有前端） |
| 6 | **交叉分析** | 合并同类问题，追溯根本原因 |
| 7 | **路线图** | 分阶段行动方案 |

### 3. 交叉分析——核心增值

多维度的发现合并，避免信息孤岛：

- 架构问题 + 代码质量问题有同源的 → 合并成一个治理项
- 依赖建议 + 架构评估有冲突的 → 给出正确的执行顺序
- 区分症状和病因（"重复代码"是症状，"没规范"是病因）

### 4. 输出全景报告

报告包含：

1. **一键摘要（3 句话）**
   - 当前最严重的 3 个问题
   - 最应该马上做的 3 件事
   - 如果不治理，半年后会怎样

2. **各维度评分**
   - 🟢 / 🟡 / 🔴 三档
   - 每维度的 Top 3 发现

3. **合并后的治理事项**（按优先级排序）
   - P0 立即：安全/数据/功能不可用
   - P1 本周：阻碍效率/可能导致问题
   - P2 本月：维护成本/可感知体验
   - P3 季度：优化/升级/一致性
   - 💡 观察：值得留意但不处理

4. **深层建议**
   - 哪些方案已被 GitHub 生态淘汰
   - 哪些 GitHub 上有更好方案
   - 哪些能力项目有潜力但还没做

5. **分阶段路线图**
   - 现在 → 中期 → 长期

## 运行方式

`/repo-guardian:full-scan` — 全维度扫描，输出完整报告

`/repo-guardian:full-scan src/components` — 只在指定范围内执行全维度扫描

如果仓库规模较大但没指定范围，优先扫核心业务模块，在报告里标注覆盖范围。不要求一次扫完所有代码。
