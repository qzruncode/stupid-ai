# Performance Audit

审查性能瓶颈：热点路径、不必要的计算、包体积、渲染效率、资源消耗。

## 工作步骤

### 1. 确定项目类型

先判断项目类型，只关心适合该类型的性能问题：

| 项目类型 | 关注重点 |
|---|---|
| Web 前端 | 包体积、渲染性能、瀑布请求、懒加载、静态资源 |
| Node 后端 | 查询效率、内存使用、IO 路径、缓存策略 |
| CLI/工具链 | 启动速度、计算效率 |
| 通用 | 复杂度、不必要的计算、内存泄漏风险 |

### 2. 代码扫描

| 类别 | 检查点 |
|---|---|
| 包体积 | 未 code splitting、import 了全量库但只用一小部分 |
| 渲染（前端） | 列表无虚拟化、昂贵计算在渲染路径中、滥用或缺失 memo |
| 数据请求 | N+1 查询、串行请求、缺少缓存 |
| 计算效率 | 循环内重复计算、O(n²) 操作在关键路径、昂贵但未缓存 |
| IO/内存 | 同步阻塞、闭包泄漏、无界增长、未清理的监听器 |
| 数据库 | 无索引查询、全表扫描、事务过大、缺连接池 |

### 3. 区分问题等级

- **严重**：用户可感知的卡顿/延迟、可能 OOM 或超时
- **重要**：在慢设备或数据量大时会成为瓶颈
- **优化项**：理论上更优但用户无感知，标为"值得关注但不紧急"

只关注**真的会影响体验**的性能问题。不要给"for 循环不如 map 快"这种微观优化建议。

### 4. 输出优化建议

每个优化项：
- **问题**：具体性能瓶颈
- **证据**：代码路径、数据引用
- **影响**：用户能感受到什么（"首屏多 300ms"、"列表滚动卡顿"）
- **方案**：怎么优化
- **预期提升**：优化后的量化效果
- **验证方式**：怎么确认优化生效

## 运行方式

`/repo-guardian:perf-audit` — 全仓库性能审计

`/repo-guardian:perf-audit src/api` — 只查指定模块

不输出和项目类型无关的性能建议。没有显著瓶颈就明确说"这个范围内暂无性能问题"。
