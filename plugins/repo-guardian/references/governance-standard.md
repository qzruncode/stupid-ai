# 仓库治理标准

`repo-guardian` 的审计、治理和路线图都遵循这份标准。根据仓库实际情况选择检查内容，不要机械执行清单。

## 证据等级

- **已确认**：已复现的行为、失败检查、实际测量结果、权威漏洞公告，或能直接证明结论的代码与数据链路。
- **强证据**：仓库内有多个一致信号，但仍缺少运行时或生产环境验证。
- **待调查**：值得继续调查的模式、代码味道或外部方案，不能当作仓库已经存在的问题。

每个发现都必须标明证据等级和准确证据。只有文件路径、没有因果链路，不算证据。工具告警只是线索，必须进一步确认它对用户或工程的实际影响。

## 治理维度

只选择与仓库和用户范围有关的维度。

| 维度 | 检查内容 |
|---|---|
| 产品正确性 | 功能流程、状态错误、数据完整性、异常处理、边界条件、协议漂移、关键行为缺少测试 |
| 安全与可靠性 | 信任边界、密钥、鉴权、注入、不安全解析、依赖漏洞、恢复机制、并发、资源泄漏、可观测性 |
| 架构 | 职责归属、模块边界、依赖方向、状态和数据流、耦合、扩展成本、部署边界 |
| 可维护性 | 重复规则、平行实现、死路径、无效抽象、复杂度、命名、可测试性、文档漂移 |
| 依赖与生态 | 停止维护或存在漏洞的包、重复依赖、框架或标准库已有能力、成熟外部方案、许可与迁移风险 |
| 性能与运维 | 用户可感知延迟、吞吐、查询、I/O、内存、CPU、包体积、启动、缓存、构建与部署瓶颈 |
| 界面与交互 | 任务完成效率、信息层级、状态反馈、一致性、响应式、无障碍、视觉质量、设计系统复用 |
| 产品演进 | 缺失能力、流程阻力、升级路径、运营准备度、差异化，以及项目应该主动停止的工作 |

不能只凭文件行数、嵌套层数、版本差距、star 数或项目年龄判定问题。这些数值只能帮助定位候选，最终必须证明真实影响。

## 仓库调查

1. 判断或修改前，先读取仓库规则并检查工作区状态。
2. 还原项目目标、用户、公开行为、技术栈、入口、模块、数据存储、外部集成、构建、测试、部署和运行面。
3. 使用全仓搜索、依赖追踪和调用链追踪验证假设。抽样必须有明确依据；只读了代表性文件时，不得声称已经覆盖全部代码。
4. 在安全的前提下运行现有检查和定向复现。区分原有失败、环境问题和当前工作引入的失败。
5. 判断界面问题时，如果项目能够运行，必须检查相关视口和交互状态下的真实页面。只看源码不能证明视觉质量。
6. 合并由同一根因造成的多个症状，不用重复表现虚增问题数量。
7. 记录已覆盖范围、跳过区域、失败检查、不可用环境和其他盲区。

## 方案检索与外部验证

外部调研分成两条路径，各自解决不同问题：

1. **找现成方案**：首选 GitHub MCP 搜索相关仓库，再读取候选仓库中与当前问题直接相关的源码。用真实实现判断它解决了什么、如何集成、是否值得复用，不能根据项目简介、star 数或搜索摘要直接推荐。
2. **核对外部事实**：使用普通网页搜索查官方文档、标准、发布记录、版本、漏洞数据库以及浏览器或运行时行为。优先采用一手来源，并给时效性结论附链接。

判断重复造轮子、寻找替代实现、比较开源方案时必须走第一条路径；核对依赖版本、漏洞、规范和兼容性时走第二条路径。两条路径可以组合，但不能互相冒充。

找到候选方案后，根据当前仓库的真实约束比较：

- 能力与 API 是否匹配；
- 维护和发布是否健康；
- 安全记录与供应链风险；
- 许可证与治理方式；
- 运行时、包体积、基础设施和运维成本；
- 迁移成本与兼容性；
- 与当前技术和场景有关的采用证据。

流行度不能证明适用性。不能仅仅因为 GitHub 上存在一个仓库，就建议替换运行正常的自研实现。必须区分：

- **重复建设**：没有仓库特定理由，却重新实现已经成熟解决的通用能力；
- **方案落后**：当前做法明显较弱，并且有证据证明另一方案更适合本仓库；
- **合理自研**：领域行为或项目约束足以支撑自主维护。

GitHub MCP 不可用时，不得声称已经充分调查现成方案；普通网页搜索不能代替仓库和源码检索。无法查询外部事实时，必须说明版本、漏洞和生态结论尚未得到当前验证。

## 优先级与决策

根据已证明的影响、发生可能性、影响范围、紧迫性、置信度、依赖顺序、成本和可逆性排序。安全、数据丢失、核心功能损坏和发布阻塞通常优先于清理。不能因为视觉调整成本低，就把它排在代价更高但影响更大的正确性问题之前。

每项建议都必须明确选择一种决策：

- **保留**：现有方案合理；
- **修复**：修正现有设计或实现；
- **替换**：迁移到更合适的内部、标准或外部方案；
- **新增**：补齐能够推进项目目标的能力；
- **推迟**：当前证据或价值不足；
- **移除**：项目不应继续承担这项能力或复杂度。

## 发现条目

调查后仍然成立的问题，使用以下紧凑结构：

- `ID` 和一句话结论；
- 决策：保留、修复、替换、新增、推迟或移除；
- 优先级和证据等级；
- 准确的仓库证据，以及必要时的外部证据链接；
- 受影响的用户或工程结果；
- 根因，而不只是表面症状；
- 推荐改法和真正有意义的备选方案；
- 影响范围、迁移或上线风险、前置依赖；
- 完成证据：测试、测量、截图、安全检查或可观察结果。

不要编造精确数字、性能收益、工作量或时间。应当实际测量，否则明确标为未知。
