# repo-guardian

> 从真实证据到实际改进，再到下一步方向。

`repo-guardian` 用一套闭环治理仓库里的任何高价值问题：先确认问题和机会，再完成修改并证明收益；需要决定项目方向时，再基于仓库目标和当前生态安排路线图。

它覆盖但不限于：架构、正确性、安全漏洞、代码质量、冗余、依赖过时、性能、可运维性、前端视觉与交互、无障碍、重复造轮子、低效技术方案、功能完善和项目演进。

## 三个 skill

| 命令 | 做什么 | 是否改代码 |
|---|---|---|
| `/repo-guardian:audit [范围或关注点]` | 还原项目全貌，验证问题，输出有证据和优先级的治理清单 | 否 |
| `/repo-guardian:improve [问题或 finding ID]` | 修复、替换、升级、移除或补齐能力，并用前后证据验收 | 是 |
| `/repo-guardian:roadmap [方向]` | 决定项目接下来该保留、停止、治理和建设什么 | 否 |

问题类型是审查维度，不是一排互相割裂的命令。三个 skill 分别负责发现、落地和演进，可以单独使用，也可以按需串联：

```text
/repo-guardian:audit 支付链路
  -> 得到已确认的 RG-003

/repo-guardian:improve RG-003
  -> 修复根因、迁移调用方、验证前后差异

/repo-guardian:roadmap
  -> 决定下一项最值得投入的工作
```

也可以直接治理一个明确问题，不必先跑全仓审计：

```text
/repo-guardian:improve 前端首屏很慢，定位并解决真正的瓶颈
/repo-guardian:improve 检查这个自研任务队列是否应该换成成熟方案
/repo-guardian:audit 登录模块的安全、可靠性和交互问题
```

## 治理原则

- **先证明，再定性**：代码味道、工具告警和流行方案只是线索，不直接算问题。
- **不仅报告，还能改完**：`improve` 必须完成实现、迁移、验证和最小防回归护栏。
- **GitHub MCP 找方案，官方资料核事实**：先从相关仓库和源码判断现成方案能否解决问题，再核对维护、安全、许可、版本和兼容性，不拿搜索摘要或 star 数替代判断。
- **不拿固定阈值制造问题**：文件长度、嵌套层数、版本差距只用于定位候选，最终看真实影响。
- **覆盖范围说清楚**：没运行起来的 UI、没覆盖的模块、没验证的线上假设都会明确标出。
- **保护已有成果**：允许结论是保留现状；不会为了显得有产出而强行重构。
