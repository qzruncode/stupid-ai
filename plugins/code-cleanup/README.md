# code-cleanup

> 用一份稳定功能基准，持续把实现改得更高效。

`code-cleanup` 把功能基准管理和代码优化彻底分开。唯一功能基准是：

```text
docs/product-spec/feature-list.md
```

它不写源码路径、函数或框架，而是完整描述角色、入口、输入校验、业务流程、异常状态、界面交互、权限、副作用、外部协议和验收场景。把这一个文件交给另一个模型，它应当能在看不到原代码的情况下复刻并验收产品。代码可以反复重写，功能正确性始终只和这份基准比较。

## 三个独立命令

| 命令 | 输入 | 唯一允许的主要输出 | 不会做的事 |
|---|---|---|---|
| `/code-cleanup:generate` | 现有代码库和运行行为 | 初始功能基准 | 不改代码，不开始优化 |
| `/code-cleanup:update` | 已确认的业务变化 | 新版本功能基准 | 不实现需求，不用当前代码定义业务 |
| `/code-cleanup:optimize` | `ready` 功能基准和现有代码 | 符合基准且有指标改善的代码 | 不修改基准，不迁就错误现状 |

三个命令不会自动串联。你可以让不同模型、不同会话甚至不同团队分别执行，交接边界始终是同一份功能清单。

## 使用

```text
/code-cleanup:generate
/code-cleanup:generate 只生成管理后台范围的完整功能清单

/code-cleanup:update F-014 的退款时限从 24 小时调整为 48 小时

/code-cleanup:optimize F-014，降低接口延迟和数据库查询次数
/code-cleanup:optimize 在不改变功能基准的前提下，重写效率最低的模块
```

`generate` 和 `update` 会使用插件自带的结构校验器，但结构正确不代表内容完整。只有没有阻塞性未知项、且仅凭清单即可实现和验收的功能，才能标记为 `ready`。

`optimize` 不把优化前代码当功能基准：代码与清单不一致时修代码；然后在相同条件下比较延迟、吞吐、CPU、内存、I/O、查询次数、构建产物、依赖或复杂度等指标。功能基准全部通过且至少一项效率证据改善，优化才算完成。

## 安装

```text
/plugin marketplace add qzruncode/stupid-ai
/plugin install code-cleanup@stupid-ai
```

## 原则

- 功能基准定义“做什么”，代码只负责“怎么高效做到”。
- 首次生成后，代码不再拥有定义功能的权力。
- 业务变化先单独更新基准，代码实现和优化属于其他流程。
- 代码与 `ready` 基准冲突时，基准胜出并修正代码。
- 优化成功必须同时证明功能符合基准和效率指标改善。
