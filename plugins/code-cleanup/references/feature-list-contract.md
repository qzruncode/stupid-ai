# 功能清单契约

`docs/product-spec/feature-list.md` 是代码库的唯一功能基准。它必须脱离原代码库独立成立：另一个模型只读取这一个文件，就能实现并验收行为等价的产品。

## 不变量

- 只描述产品、业务和外部可观察行为，不记录源码路径、函数、类、模块、数据库表、框架或“当前代码如何实现”。
- 不依赖其他文档才能理解；必要定义、规则和协议必须写进本文件。
- 精确描述事实，不使用“正常处理”“按需展示”“同现有逻辑”等无法复刻的表述。
- 保留稳定功能 ID。重命名不换 ID；删除功能时标记 `deprecated` 并说明替代关系，不把 ID 复用给其他功能。
- 功能状态只使用 `ready`、`draft`、`deprecated`。含有阻塞性未知项的功能必须是 `draft`。
- `ready` 表示该功能仅凭本文即可复刻和验收，并可直接作为代码实现、重写和优化的验收基准；不能出现 `TODO`、`TBD`、`未知`或`待确认`。
- 首次生成时可以从当前代码恢复功能事实；基准建立后，代码不再拥有定义功能的权力。
- 只有已确认的业务变化可以通过 `/code-cleanup:update` 改变基准。代码新增、遗漏或偶然实现的行为不能反向改写基准。
- 代码与 `ready` 基准冲突时，基准胜出。把冲突记为实现偏差并修正代码，不停止优化，也不修改清单迁就代码。

## 固定结构

保持以下二级标题及顺序：

1. `## 1. 文档控制`
2. `## 2. 产品边界`
3. `## 3. 参与者与权限`
4. `## 4. 术语与业务对象`
5. `## 5. 全局规则`
6. `## 6. 功能索引`
7. `## 7. 功能详述`
8. `## 8. 跨功能旅程`
9. `## 9. 外部契约总表`
10. `## 10. 数据生命周期`
11. `## 11. 质量属性`
12. `## 12. 排除项与未决事项`
13. `## 13. 变更记录`

### 文档控制

至少包含：

```markdown
- **产品**: <产品名>
- **规格版本**: <语义化版本>
- **基线日期**: <YYYY-MM-DD>
- **覆盖状态**: draft | partial | ready
- **覆盖范围**: <本文件覆盖的产品边界>
- **不包含范围**: <明确排除内容；没有则写“无”>
```

`ready` 表示整个覆盖范围均可复刻；`partial` 表示只有索引中标为 `ready` 的功能可作为重构契约；`draft` 不得用于代码优化。

### 功能索引

索引至少包含功能 ID、名称、契约状态、主要参与者、入口或触发方式、依赖功能 ID。索引必须覆盖详述中的所有功能，状态保持一致。

### 功能详述

每项使用 `### F-001 功能名称` 形式，并按以下固定小节描述。某项确实不适用时写明“不适用”及原因，不能删除小节。

```markdown
### F-001 功能名称

- **契约状态**: ready | draft | deprecated
- **目标**: <用户或业务获得什么结果>

#### 参与者与权限

#### 触发与前置条件

#### 输入与校验

#### 主流程

#### 分支与异常

#### 状态与反馈

#### 界面与交互

#### 输出与副作用

#### 业务规则

#### 外部契约

#### 非功能约束

#### 依赖

#### 验收场景
```

详述质量要求：

- 输入列出字段含义、类型或格式、必填性、默认值、边界、校验失败反馈和敏感性。
- 流程覆盖成功、空数据、加载、失败、无权限、重复操作、超时和恢复；不适用的状态明确写出。
- 界面描述信息层级、交互、可见条件、文案语义、响应式和无障碍要求；非 UI 功能说明其调用界面。
- 输出写清返回结果、持久化变化、通知、审计、计费等副作用，以及失败时是否回滚。
- 业务规则使用可判定的条件与优先级，说明时间、金额、精度、排序、去重、幂等、并发和时区语义。
- 外部契约给出公开页面地址、API、事件、Webhook、任务或 CLI 的精确输入输出、错误语义和兼容要求。无外部契约时写“不适用”。
- 非功能约束只记录复刻所需的可验收指标，包括性能、安全、隐私、可用性、容量和合规要求；证据不足时不得编造数值。
- 依赖只引用业务能力、外部系统或功能 ID，不引用实现模块。
- 验收场景使用 Given / When / Then，覆盖主流程和高风险边界。

### 系统级内容

- 产品边界说明解决的问题、成功结果、入口类型、部署或租户边界及明确不做的事情。
- 参与者与权限给出角色矩阵、身份生命周期和越权反馈。
- 术语与业务对象给出业务含义、关键属性、关系、状态机和不变量，不给数据库结构。
- 全局规则统一认证、授权、导航、错误、时间、语言、金额、隐私、审计、通知和无障碍语义。
- 跨功能旅程串联端到端目标，引用功能 ID，并描述中断、恢复和补偿。
- 外部契约总表必须与各功能详述一致，便于兼容性核对。
- 数据生命周期描述产生、读取、变更、归档、导出、删除、保留和恢复语义。
- 质量属性记录系统级可验收约束。
- 排除项与未决事项明确边界。存在影响复刻或验收的未决事项时，覆盖状态不能是 `ready`。
- 变更记录至少写版本、日期、变更类型、功能 ID、行为变化和兼容影响，不写文件 diff。

## 完整性检查

交付前执行：

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/scripts/validate_feature_list.py" docs/product-spec/feature-list.md
```

如果运行环境没有 `CLAUDE_PLUGIN_ROOT`，定位已安装的 `code-cleanup` 插件后执行同一脚本。结构校验通过不等于内容已完整，还必须进行“只给清单复刻”的人工式审读：隐藏代码库上下文，逐项确认入口、规则、状态、协议和验收场景是否足以消除实现猜测。
