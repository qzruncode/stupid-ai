# 功能清单契约

`docs/product-spec/feature-list.md` 是代码库的唯一功能基准。它必须脱离原代码库独立成立：任意 AI 只读取这一个文件，就能实现并验收清单声明范围内的完整产品。

## 不变量

- 只描述产品、业务和外部可观察行为，不描述实现。
- 不包含源码、伪代码、代码块、行内代码格式、文件或目录路径、实现符号、内部结构、框架、依赖、配置、环境变量、数据库结构或实现证据。
- 不包含内部路由模板、权限码、配置键、原始状态常量、国际化键或代码风格标识符。公开能力用产品语言描述，不泄露内部命名。
- 不依赖其他文档；复刻产品所需的定义、体验、规则、公开协议和验收结果全部写进本文件。
- 不使用“正常处理”“按需展示”“参考现有逻辑”等需要实现者猜测的表述。
- 保留稳定功能 ID。重命名不换 ID；废弃功能标记 `deprecated` 并说明替代关系；不复用 ID。
- 功能状态只使用 `ready`、`draft`、`deprecated`。影响复刻的内容未确定时必须是 `draft`。
- `ready` 表示仅凭本文即可复刻和验收，并可直接作为实现、重写和 cleanup 的唯一验收基准；不能出现 `TODO`、`TBD`、`未知`或`待确认`。
- 首次初始化可以从现有产品恢复功能事实；基准建立后，代码不再拥有定义功能的权力。
- 只有已确认的业务变化可以通过 `/code-cleanup:feature-list` 更新基准。代码变化不能反向改写基准。
- 代码与 `ready` 基准冲突时，基准胜出。通过 `/code-cleanup:cleanup` 修正代码，不修改清单迁就实现。
- 未明确指定子范围时，基准覆盖仓库承载的整个可交付产品；用户明确指定 CodeBench 等产品范围时，以该声明范围为边界，不擅自扩大或缩小。
- 覆盖状态相对于“覆盖范围”判定。显式子范围可以在范围内全部可复刻时成为 `ready`；`partial` 表示声明范围尚未盘点完整或仍含阻塞内容，只是可恢复断点。

## 固定结构

保持以下二级标题及顺序：

1. `## 1. 文档控制`
2. `## 2. 产品边界`
3. `## 3. 参与者与权限`
4. `## 4. 术语与业务对象`
5. `## 5. 产品体验规范`
6. `## 6. 全局规则`
7. `## 7. 功能索引`
8. `## 8. 功能详述`
9. `## 9. 跨功能旅程`
10. `## 10. 外部契约总表`
11. `## 11. 数据生命周期`
12. `## 12. 质量属性`
13. `## 13. 排除项与未决事项`
14. `## 14. 变更记录`

### 文档控制

至少包含产品、规格版本、基线日期、覆盖状态、覆盖范围和不包含范围，每项使用 `- **字段**: 值` 格式。规格版本使用语义化版本，基线日期使用 `YYYY-MM-DD`；覆盖状态只使用 `draft`、`partial` 或 `ready`。

`ready` 表示声明的整个覆盖范围均可复刻；`partial` 表示声明范围尚未完整，只有索引中标为 `ready` 的功能可作为局部验证基准。cleanup 可以处理与 `draft` 完全隔离的 `ready` 功能候选，但依赖未决语义的候选必须阻塞，且整个声明范围不得标为清理完成。

### 功能索引

索引至少包含功能 ID、名称、基准状态、主要参与者、入口或触发方式、依赖功能 ID。索引必须覆盖详述中的所有功能，状态保持一致。

### 功能详述

每项使用 `### F-001 功能名称` 形式，随后使用 `- **基准状态**: ready` 和 `- **目标**: ...` 两行，再依次包含：参与者与权限、触发与前置条件、输入与校验、主流程、分支与异常、状态与反馈、界面与交互、输出与副作用、业务规则、外部契约、非功能约束、依赖、验收场景。基准状态可改为 `draft` 或 `deprecated`。确实不适用时写明“不适用”及原因，不能删除小节。

详述质量要求：

- 输入列出业务含义、类型或格式、必填性、默认值、边界、校验失败反馈和敏感性。
- 流程覆盖成功、空数据、加载、失败、无权限、重复操作、超时、中断和恢复；不适用的状态明确写出。
- 界面描述页面结构、信息层级、准确文案、可见条件、操作、反馈、加载与空状态、响应式和无障碍要求；非 UI 功能说明使用者如何触发并观察结果，不描述内部调用方式。
- 输出写清用户获得的结果、业务数据变化、通知、审计、计费等副作用，以及失败时是否回滚。
- 业务规则使用可判定条件和优先级，说明时间、金额、精度、排序、去重、幂等、并发和时区语义。
- 外部契约给出公开页面、API、事件、Webhook、任务、CLI 或文件的精确输入输出、错误语义和兼容要求。使用自然语言和表格，不提供代码样例或内部设计。
- 非功能约束只记录可验收的产品指标，包括性能、安全、隐私、可用性、容量和合规要求；没有权威数值时不编造。
- 依赖只引用业务能力、外部系统或功能 ID，不引用实现模块。
- 验收场景使用 Given / When / Then，覆盖主流程和高风险边界。

### 系统级内容

- 产品边界先列出声明范围内的全部可交付应用、服务和产品域及其覆盖状态，再说明解决的问题、成功结果、入口类型、租户或交付边界及明确不做的事情。未指定范围时，这里的声明范围就是仓库承载的整个可交付产品。
- 参与者与权限给出角色矩阵、身份生命周期和越权反馈。
- 术语与业务对象给出业务含义、关键属性、关系、状态机和不变量，不给数据库结构。
- 产品体验规范给出信息架构、全局导航、页面清单、布局层级、品牌与视觉规则、排版、颜色、间距、图标与媒体语义、通用控件、反馈模式、准确文案原则、响应式和无障碍要求，使另一个 AI 能复刻完整产品体验。
- 全局规则统一认证、授权、导航、错误、时间、语言、金额、隐私、审计和通知语义。
- 跨功能旅程串联端到端目标，引用功能 ID，并描述中断、恢复和补偿。
- 外部契约总表与各功能详述一致，便于兼容性核对。
- 数据生命周期描述产生、读取、变更、归档、导出、删除、保留和恢复的业务语义。
- 质量属性记录系统级可验收约束。
- 排除项与未决事项明确边界。存在影响复刻或验收的未决事项时，覆盖状态不能是 `ready`。
- 变更记录至少写版本、日期、变更类型、功能 ID、行为变化和兼容影响，不写文件 diff。

## 完整性检查

交付前运行 `python3 "${CLAUDE_PLUGIN_ROOT}/scripts/validate_feature_list.py" docs/product-spec/feature-list.md`。如果运行环境没有 `CLAUDE_PLUGIN_ROOT`，定位已安装插件后执行同一脚本。

结构校验通过不等于内容完整。必须隐藏原仓库和所有其他文档做盲复刻审读，逐项确认产品范围、产品体验、入口、角色、规则、状态、数据语义、公开协议和验收场景是否足以消除实现猜测。最终文件中出现任何实现内容都视为失败。
