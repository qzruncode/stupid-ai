# Cleanup Plan Contract

`docs/code-cleanup/cleanup-plan.md` 是 cleanup 的技术执行状态，不是功能基准。它可以记录代码路径、实现证据和验证命令；任何功能含义仍只由 `docs/product-spec/feature-list.md` 决定。

## 固定结构

保持以下二级标题及顺序：

1. `## 1. 运行控制`
2. `## 2. 范围与排除项`
3. `## 3. 文件覆盖表`
4. `## 4. 候选队列`
5. `## 5. 功能基准验证`
6. `## 6. 批次记录`

### 运行控制

每项使用 `- **字段**: 值` 格式，至少包含：

- 状态：只使用 `inventory`、`executing`、`blocked`、`complete`。
- 功能基准版本：来自功能清单。
- 范围基准状态：只使用 `ready` 或 `blocked`。只有清理范围关联的功能全部为 `ready` 时才写 `ready`；`blocked` 不妨碍执行与缺口完全隔离的安全候选，但禁止整个计划完成。
- 清理范围：用户本次明确指定时写该范围，否则继承功能清单的“覆盖范围”；不得默认扩成完整仓库或根据当前改动缩小。
- 扫描基线：首次扫描所依据的版本标识。
- 扫描终点：最近纳入扫描的版本标识。
- 总文件数：范围内应审查的源码、测试和工程配置文件数量。
- 已审文件数：状态不为 `unreviewed` 的文件数量。
- 覆盖率：已审文件数除以总文件数，写为百分比。

### 范围与排除项

列出纳入的应用、包、服务、语言、生成物边界和所有排除项。只能排除生成物、第三方代码、锁文件、构建产物或用户明确排除的范围；不能因为目录太大而跳过。

### 文件覆盖表

使用 Markdown 表格，列固定为：`路径 | 功能 ID | 状态 | 候选 ID | 说明`。

每个范围内文件必须恰好出现一次。状态只使用：

- `unreviewed`：尚未完成语义审查。
- `reviewed-clean`：已审查，未发现值得处理的问题。
- `candidate`：关联一个或多个候选。
- `blocked`：因功能基准或外部决策无法审查。
- `removed`：已由 cleanup 删除。

### 候选队列

使用 Markdown 表格，列固定为：`候选 ID | 类别 | 功能 ID | 代码位置 | 证据 | 规范方向 | 风险 | 验证 | 状态`。

候选 ID 使用 `C-001` 形式并保持稳定。状态只使用：

- `candidate`：证据仍需补齐。
- `ready`：边界、收益和验证已明确，可以执行。
- `active`：正在执行。
- `done`：实现与验证完成。
- `blocked`：需要功能基准或外部决策。
- `declined`：经明确判断不值得修改，并记录原因。

候选至少覆盖：重复业务规则、分叉实现、过大文件或职责、过度抽象、透传包装、冗余状态、死代码、兼容残留、不一致模式、依赖问题、自制基础设施、算法、I/O、请求、渲染、错误处理、权限、国际化、测试与工程护栏。

### 功能基准验证

记录每个目标功能 ID 的验证方式、清理前结果和最新结果。验证必须面向功能清单，不得把旧代码行为当基准。

### 批次记录

按时间记录每个执行批次、候选 ID、实际修改、删除路径、验证结果、收益证据和下一批次。单次删除未引用文件只能关闭对应候选，不能关闭整个计划。

## 完成门禁

只有同时满足以下条件才能把状态改为 `complete`：

- 功能清单对清理范围全部为 `ready`。
- 文件覆盖率为 100%，文件覆盖表没有 `unreviewed` 或 `blocked`。
- 所有候选均为 `done` 或有充分理由的 `declined`；不存在 `candidate`、`ready`、`active` 或 `blocked`。
- 所有目标功能验收、受影响测试和项目质量检查通过。
- 批次记录包含清理前后收益证据和防回退护栏。

未满足时保持 `inventory`、`executing` 或 `blocked`，并明确报告 cleanup 尚未完成。交付前运行插件的 cleanup plan 校验器。

校验命令：`python3 "${CLAUDE_PLUGIN_ROOT}/scripts/validate_cleanup_plan.py" docs/code-cleanup/cleanup-plan.md`。如果运行环境没有 `CLAUDE_PLUGIN_ROOT`，定位已安装插件后执行同一脚本。
