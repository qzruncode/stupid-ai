# Code Cleanup

把代码库收敛成更小、更清晰、可继续迭代的系统，同时保持现有行为。不要把删除行数当目标；目标是减少重复实现、分叉规则和维护同一行为所需理解的概念数量。

## 持续产物

在目标项目维护两个版本化文件；已有同类文档时复用，不创建平行体系。

- `docs/code-health/feature-checklist.md`：页面、接口、任务、CLI 和关键内部能力的活清单。每项记录稳定 ID、入口、用户可观察行为、输入与前置条件、成功/空/错误/权限状态、实现证据、验证方式，以及 `verified`、`inferred` 或 `unknown` 置信度。
- `docs/code-health/cleanup-ledger.md`：候选收敛批次及其目标、证据、关联行为 ID、规范实现归属、风险、状态、验证结果和实际减少的重复实现。状态只使用 `candidate`、`ready`、`active`、`blocked`、`done`、`declined`。

不得根据命名猜测功能。优先从路由、控制器、公开类型、调用链、测试、产品文档和运行结果提取行为；证据不足时标为 `inferred` 或 `unknown`。每次运行只补齐本次触达范围及其相邻依赖，不因全局清单尚未完成而阻塞已被覆盖的安全批次。

## 执行闭环

### 1. 恢复项目全貌

- 读取项目规则、架构文档、现有测试与上述两个持续产物。
- 建立主要目录、应用入口、页面路由、API、后台任务、数据模型、状态管理、共享库、构建和部署的轻量地图。
- 首次运行广泛盘点；后续运行先读取账本，核对上次之后的代码变化，增量更新触达的行为。
- 记录明确跳过的范围，不能把部分覆盖表述成全量完成。

### 2. 建立行为安全网

- 把当前实际存在的行为写入 feature checklist，而不是把期望功能误写成现状。
- 先复用现有测试。对将被改动但缺少可靠验证的高风险行为，补最小的 characterization test 或可重复检查。
- 一个批次只有在其所有关联行为均有证据和可执行验证时才标为 `ready`。
- 遇到互相矛盾的现状、无法从代码判断的产品意图、公开契约变化、数据迁移或不可逆操作时，标为 `blocked` 并请求人类决策；其余阶段自动继续。

### 3. 找到真正的收敛点

同时检查文本重复和语义重复，重点寻找：

- 同一业务规则、格式化、校验、权限、请求、缓存或错误处理的多套实现。
- 只因 Agent 未找到现有能力而新增的 helper、hook、service、组件、模型或适配层。
- 已无调用的代码、透传包装、重复状态、冗余数据转换和不再成立的兼容层。
- 用自制框架重复成熟依赖已可靠提供的能力。
- 已无法局部修复、但外部行为可以被明确提炼的模块。

每个候选批次必须给出具体证据、受影响调用方、拟保留的规范实现、关联行为 ID 和风险。按用户价值、重复扩散速度、修改确定性和验证强度排序；不要仅按代码行数排序。

替换自制实现前，先确认现有依赖中是否已有能力；引入新依赖时再核对维护状态、许可证、运行环境、体积、安全、迁移成本和锁文件影响。收益不明确时保留现状。

### 4. 按语义批次精简

每次只处理一个边界清晰的重复簇：

1. 运行批次前验证并记录基线。
2. 选定唯一规范实现和归属模块。
3. 迁移全部已知调用方，保持公开契约和用户可观察行为不变。
4. 删除被替代实现、无用包装和过期测试，不保留“以防万一”的平行路径。
5. 运行关联行为检查、受影响模块测试和项目已有质量检查。
6. 更新 feature checklist 与 cleanup ledger，再进入下一个 `ready` 批次。

如果模块需要重写，先冻结其输入、输出、副作用、错误语义、权限和关键性能约束，并用 characterization tests 固定这些行为；在稳定边界后实现替代模块，对比验证通过后再切换调用方并删除旧实现。不能证明等价时停止，不以“代码更漂亮”为由改变行为。

不要覆盖用户未提交的无关改动，不顺手扩大重构范围，不为了统一风格改写无关文件。除非用户明确要求，不自动提交或推送代码。

### 5. 固化防回退护栏

完成首个有效批次后，结合项目已有工具补上最小可执行护栏：

- 在现有 `CLAUDE.md` 或等价 Agent 规则中加入：先搜索再创建；同一业务规则只有一个规范归属；修改用户可观察行为时同步 feature checklist；新增平行实现必须说明原因。
- 在现有 lint、测试和 CI 中优先复用可执行检查，覆盖架构边界、循环依赖、未使用代码、重复实现或项目特有禁用模式。
- 对暂时无法一次清零的存量问题采用 baseline ratchet：允许当前基线，禁止新增或恶化；随着批次完成只收紧基线。
- 让 CI 失败信息指出规范实现或修复入口，避免只给出抽象分数。

不要为了声称“有护栏”而安装一堆不适配项目的工具。任何新增检查都必须在本地真实运行，并控制误报。

## 运行方式

`/code-cleanup:cleanup` 自动识别当前处于首次盘点、增量补图、候选排序、批次执行还是护栏固化阶段，恢复账本后尽可能持续推进。用户指定模块时只治理该范围及必要依赖；未指定时先建立全局地图，再完成最高价值且验证充分的批次。

如果本次调用伴随普通功能开发，先更新触达行为，完成开发后检查是否新增了平行实现，并同步清单与账本；不重新全量扫描无关模块。

结束时简要报告：清单覆盖变化、完成的收敛批次、保留的规范实现、删除的重复实现、验证证据、已安装护栏、剩余最高优先级候选和真实阻塞项。代码行数只能作为附带指标，不能作为成功标准。
