# Optimize Against Functional Baseline

以 `docs/product-spec/feature-list.md` 为唯一功能基准优化代码。允许重组或彻底重写实现，但不得修改功能清单；最终代码必须满足目标范围内全部 `ready` 功能，并用前后证据证明实现效率得到改善。

## 开始前

完整读取 `../../references/feature-list-contract.md`，再读取功能清单并运行结构校验。

只有以下情况阻塞目标范围：

- 功能清单不存在。
- 目标功能不是 `ready`，或者基准内部自相矛盾，无法形成唯一验收结果。
- 优化涉及不可逆数据操作，而基准没有定义所需的数据保留、迁移或恢复语义。

当前代码与基准冲突不是阻塞项。把它记为实现偏差，在优化中把代码修到基准。代码里存在但基准没有定义的行为不具备基准地位，不能反过来限制或改写优化目标。

## 优化闭环

1. **锁定功能基准**：确定目标功能 ID，从清单提取角色、输入、流程、状态、权限、输出、副作用、外部协议、非功能约束和验收场景，形成基准验收矩阵。
2. **审计当前实现**：把代码、调用方和现有测试映射到验收矩阵，列出缺失、错误和额外行为。缺失或错误行为必须修到基准；现有测试与基准冲突时修测试，不保留错误现状。
3. **建立优化前指标**：按目标选择可重复测量的指标，例如延迟、吞吐、CPU、内存、I/O、请求或查询次数、构建时间、产物体积、依赖数量、重复实现数量、圈复杂度和维护触点。没有可靠测量方法时，先补最小 benchmark、profile 或静态检查。
4. **选择高价值改造**：优先处理算法和数据结构低效、重复计算、串行 I/O、N+1 请求、无效渲染、重复规则、分叉状态、冗余转换、过度抽象、无用依赖和不必要的自制基础设施。按实际瓶颈和收益排序，不按删行数排序。
5. **重写实现**：在基准边界内自由选择更高效的结构，迁移全部调用方，删除旧路径和重复实现。不得保留两套实现来规避切换，也不得把当前代码细节写回功能清单。
6. **验证功能基准**：运行基准验收矩阵、受影响测试和公开协议检查。只要一个目标行为不符合清单，本批次就没有完成。
7. **验证效率收益**：用相同条件重测优化指标，给出前后数值；结构性优化无法用运行指标表示时，给出可复查的静态证据。没有改善、引入明显退化或证据不足时，继续调整或撤销该方案。
8. **固化护栏**：在现有测试、benchmark、lint 或 CI 中保留最小防回退检查，使后续实现既不能偏离功能基准，也不能重新引入已消除的低效路径。
9. **检查写入边界**：确认功能清单零修改、范围外用户改动未被覆盖、旧实现无残留；除非用户明确要求，不提交、不推送、不部署。

## 判定成功

- 功能正确性的唯一比较对象是功能清单，不是优化前代码。
- 目标范围所有 `ready` 验收场景通过，代码偏差已修正。
- 至少一个明确效率指标改善，其他关键指标没有不可接受的退化。
- 实现可以完全不同，但不得改变基准规定的任何用户可观察结果。
- 代码行数只能作为附带信息，不能代替功能验证和效率证据。

结束时报告：目标功能 ID、发现并修正的基准偏差、采用的优化、删除的低效或平行路径、优化前后指标、功能验收证据、效率护栏和剩余瓶颈。不得在本流程中更新功能清单。
